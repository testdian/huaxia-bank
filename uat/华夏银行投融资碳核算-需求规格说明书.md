## 1. 引言

### 1.1 编写目的

本文档依据业务讨论稿，将华夏银行投融资碳核算的**业务流程**与**系统功能清单**整理为可评审、可拆分开发的需求规格说明，供产品、研发、测试及业务方对齐范围与验收标准。

### 1.2 项目背景

华夏银行需开展投融资组合碳排放核算，满足监管报送与内部管理分析需求。本期碳核算相关能力以**功能模块形式嵌入现有绿金系统**，不单独建设独立系统；形成「总行定规则、分行做组织、客户经理补信息、系统做计算」的主线模式。

**集成约束（已确认）：** 审批、组织权限、待办待审等能力**直接沿用绿金系统既有模块**，碳核算侧仅需提供「提交审核」及业务数据回写，**不另行设计审核模块**。

### 1.3 建设原则（业务主线）

| 原则 | 说明 |
|------|------|
| 规则集中 | 总行确定核算年度、范围、行业口径及输出目标 |
| 组织协同 | 分行承担任务统筹、初审与补数组织，不宜由总行逐笔催数 |
| 责任清晰 | 客户经理负责「提供信息与材料」，不负责定义核算口径 |
| 计算自动化 | 系统承担方法匹配、参数调用、排放量计算与汇总，减少人工误差 |
| 口径可配置 | 排放因子、归因规则、方法优先级等须可配置并留痕 |
| 监管与管理分离 | 前端可计算 8+15 行业全量；监管输出时按人行要求范围提取 |
| 嵌入集成 | 碳核算能力作为绿金系统子模块交付，复用宿主系统的审批、权限、待办等通用能力 |

### 1.4 术语与缩写

| 术语 | 说明 |
|------|------|
| 8 大行业 | 监管口径下的重点行业分类（具体以人行/行内映射为准） |
| 8+15 行业 | 8 大行业 + 扩展 15 个行业，用于管理分析全量口径 |
| 候选清单 | 系统按规则从业务数据中筛选出的初步待核算范围 |
| 正式清单 | 管理端确认并锁定后的待核算业务清单 |
| 补数任务 | 针对正式清单中缺失字段向分行/客户经理派发的填报任务 |
| 归因排放 | 按银行融资占比归因后的碳排放量 |
| 非项目法 / 项目法 | 投融资碳核算的两种核算路径（由业务类型与数据情况决定） |

### 1.5 用户角色

| 角色 | 职责概要 |
|------|----------|
| 总行绿金/管理部门 | 发起任务、确认口径与正式清单、提交审核、监管输出 |
| 分行绿金负责人/分支机构管理人员 | 接收任务、组织补数、完整性初检、提交审核（按既有审批节点） |
| 客户经理/业务人员 | 按任务补录字段、上传附件材料 |
| 系统管理员 | 字典、映射、因子与参数配置（组织权限沿用绿金系统） |
| 系统自动处理 | 清单筛选、任务派发、方法匹配、计算、汇总、异常提示 |

## 2. 总体描述

### 2.1 系统定位

投融资碳核算能力以**嵌入式子模块**方式建设于现有**绿金系统**内，覆盖：**核算任务管理 → 业务识别与清单 → 核算对象与边界 → 补数协同 → 方法与参数 → 碳排放计算 → 提交审核（对接既有审批）→ 结果输出与报告**，并配套主数据、业务留痕等能力。

**不在本期碳核算模块内建设的能力：**

| 能力 | 处理方式 |
|------|----------|
| 审批流（分行初审、总行终审、节点配置、待办待审） | **沿用绿金系统既有审批模块**，碳核算模块仅提供「提交审核」入口及审批状态回写 |
| 组织树、角色权限、数据权限隔离 | **沿用绿金系统**，嵌入模块按宿主权限控制数据可见范围 |
| 审核模块 UI、流程引擎、催审规则 | **不设计、不开发**，以宿主系统为准 |

### 2.2 与外部系统关系

| 外部系统/数据源 | 关系说明 |
|----------------|----------|
| 主生产系统 / 信贷系统 | 贷款台账、客户基础信息、组织信息；接口或模板导入 |
| 数据中台（如有） | 可作为贷款台账等数据的统一接入层 |
| 公开数据接口 | 碳排放等外部数据，用于补数前预填充 |
| 原绿金系统（宿主） | **审批流、组织权限、待办待审直接复用**；碳核算模块通过接口/页面嵌入与之联动 |

**数据接入策略（讨论稿倾向）：**

- 接口对接主生产系统，**全量接入绿金系统**，在绿金系统内按规则筛选（待最终确认：是否在主系统预筛后再接口）。
- **保留 Excel 模板导入/导出** 作为接口未打通时的兜底。
- 接口可获取各业务营收数据（**贴现、保理业务除外**，该类业务缺金额需客户经理补录）。

### 2.3 功能架构（一级模块）

```
┌──────────────────────────────────────────────────────────────────┐
│                     绿金系统（宿主）                              │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │           投融资碳核算嵌入式模块（本期建设）                 │  │
│  │  核算任务 │ 业务识别与清单 │ 核算对象与边界 │ 补数采集    │  │
│  │  方法与参数 │ 碳排放计算 │ 结果输出 │ 报告 │ 数据接入映射  │  │
│  │  【提交审核】← 仅对接入口，不建设审批流                      │  │
│  └────────────────────────────────────────────────────────────┘  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │ 既有审批模块  │  │ 组织/权限    │  │ 待办、消息、操作日志  │  │
│  │ （沿用）      │  │ （沿用）      │  │ （沿用）              │  │
│  └──────────────┘  └──────────────┘  └──────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.4 与既有审批模块的集成要求

碳核算模块在业务节点完成后，须支持将以下对象**提交至绿金系统既有审批流**（具体单据类型、节点规则以宿主系统配置为准）：

| 可提交对象（示例） | 触发时机 | 碳核算模块职责 | 宿主审批模块职责 |
|-------------------|----------|----------------|------------------|
| 正式待核算清单 | Step 3 确认后 | 发起「提交审核」、展示审批状态 | 流转、会签、退回、终审、待办 |
| 补数任务包 / 填报结果 | Step 5/6 完成后 | 同上 | 同上 |
| 核算结果 | Step 8 计算完成后 | 同上 | 同上 |

**集成接口能力（碳核算侧最小集）：**

1. **提交审核**：传入业务单据 ID、单据类型、提交人、所属机构等；
2. **状态同步**：接收审批中 / 已通过 / 已退回 / 已撤回等状态，驱动业务单据锁定或解锁；
3. **退回处理**：审批退回后，碳核算模块开放对应环节修改（补数、重算等），修改后可再次提交；
4. **不在碳核算模块内实现**：审批节点配置、审批人指定、审批意见录入、待办列表、流程图等（均由宿主系统提供）。

## 3. 业务流程需求

业务流程共 **10 个步骤**，对应系统能力与关键输出如下。

### Step 1：总行发起核算任务

| 项目 | 内容 |
|------|------|
| 责任角色 | 总行绿金/管理部门 |
| 主要动作 | 确定核算年度、核算范围、适用行业；发起任务；设定组织范围；明确输出目标（监管报送/内部分析） |
| 系统支持 | 建立核算任务；选择组织范围；推送任务通知 |
| 关键输出 | 核算任务、组织范围、输出目标 |
| 管理要点 | 总行及各地分行均可发起任务 |

**待讨论：** 组织范围是否细化到支行；境外网点是否纳入。

---

### Step 2：系统识别候选核算业务清单

| 项目 | 内容 |
|------|------|
| 责任角色 | 系统自动处理 |
| 主要动作 | 从生产系统导入业务清单；按行业、业务类型、时间、余额等筛选；形成候选范围 |
| 系统支持 | 业务清单导入/接口；筛选规则引擎；候选清单生成 |
| 关键输出 | 候选核算业务清单 |
| 管理要点 | ① 接口对接主系统全量接入后在绿金筛选；② 保留模板导入导出；③ 接口可拿营收（除贴现保理） |

**待讨论：** 主系统预处理后接口 vs 全量接入后绿金处理；8+15 行业映射是否统一明确。

---

### Step 3：总行/管理端确认核算口径

| 项目 | 内容 |
|------|------|
| 责任角色 | 总行绿金/管理部门（发起人确认，含总行/分行发起场景） |
| 主要动作 | 确认核算对象是主体还是项目；明确边界与周期；确认本期纳入范围；明确待补采数据项 |
| 系统支持 | 核算对象配置；边界与周期配置；正式清单确认 |
| 关键输出 | 正式待核算清单、核算对象清单、待补数任务清单 |
| 管理要点 | 「业务识别」→「正式核算」的关键转换环节 |

---

### Step 4：系统派发补数任务

| 项目 | 内容 |
|------|------|
| 责任角色 | 系统自动处理 |
| 主要动作 | 根据正式清单生成补数任务；先尝试外部接口数据，再按组织/客户/项目/字段派发；设截止时间与提醒 |
| 系统支持 | 补数任务生成；任务派发；催办提醒 |
| 关键输出 | 补数任务列表、待填报字段清单 |
| 管理要点 | 客户经理不判断「要不要填」，只接收明确任务后补数 |

**待讨论：** 「分行接任务、客户经理协同补数」模式是否接受；本期是否开展数据补录还是先用既有数据。

---

### Step 5：分行组织补充数据

| 项目 | 内容 |
|------|------|
| 责任角色 | 分行绿金负责人/分支机构管理人员 |
| 主要动作 | 接收总行任务；统筹填报；汇总客户经理反馈；完整性初步检查 |
| 系统支持 | 分行任务看板；任务分发与汇总；完整性校验 |
| 关键输出 | 分行补录数据、初步校验后的数据包 |
| 管理要点 | 分行承担组织协调与初审，不宜总行逐笔催数；**数据补录在绿金系统进行**（倾向） |

**待讨论：** 是否允许线下 Excel 补录（影响分行数据隔离）；五种核算方法如何展示与下发（全罗列 vs 总行建议后下发）。

---

### Step 6：客户经理/业务人员补录数据与材料

| 项目 | 内容 |
|------|------|
| 责任角色 | 客户经理/业务人员 |
| 主要动作 | 按任务补录客户/项目信息；上传报告与佐证；补齐主系统缺失字段 |
| 系统支持 | 在线补录；附件上传；字段校验 |
| 关键输出 | 补充字段、附件材料、客户/项目补充信息 |
| 管理要点 | 客户经理只负责提供信息；**五种方法均展示，由客户经理选择填写方式** |

**待讨论：** 贴现、保理无信贷系统金额，需派发给客户经理处理。

---

### Step 7：系统匹配核算方法与参数

| 项目 | 内容 |
|------|------|
| 责任角色 | 系统自动处理（必要时管理端确认） |
| 主要动作 | 按业务类型与数据情况匹配方法；判断非项目法/项目法；获取排放因子、归因因子；缺失数据回退或提示 |
| 系统支持 | 方法优先级配置；自动匹配；参数库调用 |
| 关键输出 | 方法匹配结果、参数调用结果、计算准备数据集 |
| 管理要点 | 系统预置方法优先级，必要时人工确认，不建议完全手工判断 |

**待讨论：** 与联合赤道讨论数据质量验证阈值设定。

---

### Step 8：系统计算排放量并生成结果

| 项目 | 内容 |
|------|------|
| 责任角色 | 系统自动处理 |
| 主要动作 | 单笔/批量计算；多层级汇总；记录过程与结果版本；异常提示 |
| 系统支持 | 排放量计算；汇总；版本管理；异常提示 |
| 关键输出 | 排放量结果、汇总结果、计算日志、异常提示 |
| 管理要点 | 系统承担核心计算；**因子须可配置** |

---

### Step 9：提交审核（沿用绿金既有审批）

| 项目 | 内容 |
|------|------|
| 责任角色 | 业务发起人（分行/总行，按任务发起方）；审批人由**绿金系统既有审批模块**按配置执行 |
| 主要动作 | 在清单确认、补数完成或计算完成后，由业务人员在碳核算页面点击「提交审核」；审批人在宿主系统待办中处理；通过后结果锁定，退回则回到对应环节修正 |
| 系统支持（碳核算模块） | **提交审核**按钮/接口；审批状态展示；退回后业务解锁与再提交 |
| 系统支持（宿主系统） | 审批流转、分行初审/总行终审（如有）、意见填写、待办提醒——**全部沿用，不重复建设** |
| 关键输出 | 审批通过后的确认核算结果；审批记录以宿主系统为准 |
| 管理要点 | 审批节点、是否分行必审、提交时点等**以绿金系统现有配置为准**；碳核算模块不单独维护审批规则 |

**本期范围说明：** 不设计独立「审核与质量评价」模块；若需数据质量结论，可在计算环节做规则提示（见 F044），质量评分类能力（原 F049–F051）列为可选增强，与审批解耦。

---

### Step 10：结果汇总、监管提取与报告输出

| 项目 | 内容 |
|------|------|
| 责任角色 | 总行绿金/管理部门 + 系统 |
| 主要动作 | 全量汇总；提取监管范围；生成监管/管理报表；导出报告 |
| 系统支持 | 监管范围提取；管理报表；报告导出 |
| 关键输出 | 监管范围结果、内部分析结果、报表/报告文件 |
| 管理要点 | 8+15 全量计算，监管输出提取人行范围；**保留导出报告、Excel**；华夏提供上报模板；支持多维度自定义表单导出 |

**待讨论：** 报告/清单导出范围；可视化看板一期是否建设（讨论稿倾向：表格即可，复杂看板可不保留）。

## 5. 非功能需求

### 5.1 性能与容量

- 支持批量排放量计算（F040），满足年度全行客户/项目量级。
- 接口接入支持按计划频率执行，并记录每次数据量（F074）。

### 5.2 安全与权限

- 数据权限、角色权限**直接沿用绿金系统**（F058–F060），嵌入模块不得绕过宿主权限校验。
- 碳核算页面与接口须校验当前登录用户的组织与角色（由宿主系统提供）。

### 5.3 可追溯与审计

- 碳核算模块内：清单版本（F016）、计算日志（F045）、结果版本（F043）、配置变更留痕（F064）、业务操作日志（F063）须完整保留。
- 审批过程、审批意见、审批人记录以**绿金系统审批模块**留痕为准，碳核算模块仅同步审批状态。

### 5.6 集成约束

- 碳核算为**嵌入式交付**，须符合绿金系统前端框架、单点登录、菜单挂载规范。
- 「提交审核」须调用宿主系统标准审批 API/组件，禁止在碳核算模块内复制一套审批引擎。

### 5.4 可用性与运维

- 接口失败须可重跑、人工补采及 Excel 兜底（F075）。
- 任务催办、超期提醒（F021）保障填报时效。

### 5.5 兼容性

- 审批能力**完全复用**现有绿金系统，碳核算仅实现 F046 提交审核对接。
- 监管报表模板由华夏银行提供，系统支持按模板导出（F053、F057）。
